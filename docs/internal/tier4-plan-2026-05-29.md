# Tier 4 cmp_lua panel expansion — 语言特性覆盖

W1-W20 偏算法 / numeric / list / dict / closure 微 codegen。Tier 4 用一组短小的 production-shape workload 补齐 Relon 语法特性域，每条都对应至少一个 fixtures/*.relon showcase 中存在但 panel 缺测的 hot path。

## 0. 优先级分组

| 组 | 特性 | 影响面 | 实现难度 |
|---|---|---|---|
| **A. 高频运行时分派** | `match {Type: ..., *: ...}` / `#brand` 运行时 / `&ref` 解析 | 触动 tree-walker dispatch + path-cache + brand-tag 比较，分配器热路径 | 中 — production source 直接用 fixtures 现有形态 |
| **B. 容器构造糖** | dict spread / list spread / list comprehension / pipe `\|` | codegen lowering 路径与 `.map().filter()` 不同；spread 是 dict copy 主成本 | 低 — 单 production form |
| **C. 字符串 / 格式化** | f-string 插值 / split / join / replace | 与 W3 string_concat 互补；f-string 是 log/error path 主成本 | 中 — 需要 stdlib 模块支持 |
| **D. 数值 / 字面量边角** | float div/mod 混合 / hex / bin 字面量 / Int↔Float 转换 | LLVM AOT codegen 覆盖；常被常量折叠掩盖，需 black_box 拒折叠 | 低 |
| **E. Stdlib 非 list** | std/dict / std/string / std/math 模块 | 模块 import + 函数派发覆盖（目前 13/15 row 都只用 std/list） | 低 — 直接换 import 名 |
| **F. 严格模式 / 空值** | `#strict` 默认模式 / null + `??` null-coalesce | strict 模式分析器路径与 `#unstrict` 不同；null path 是 JIT 类型护栏前提 | 中 |

A 组最值得做，因为现 panel 完全没测 brand/match/ref 这套 Relon 区别于 Lua 的核心机制，导致"Relon 比 LuaJIT 快"只在算法 kernel 上成立。B/E 组其次。C/D/F 可作为后续延伸。

## 1. 待加 workload 清单

### W21_match_dispatch — `match {Brand: ...}` 运行时分派

**目的**：测 brand-tagged value 走 match 分支 + brand label 比较的成本。fixtures/polymorphism.relon 是直接 source。

**Production source (Relon)**：
```relon
#relaxed
#schema Image { name: String, url: String },
#schema Text { name: String, content: String },

#main(Int n) -> Int
range(n).reduce(0, (acc, i) => acc + classify(items[i % 2]))
where {
  items: [
    #brand Image { name: "img", url: "http://a.png" },
    #brand Text { name: "txt", content: "hello" }
  ],
  classify(it): it match {
    Image: 1,
    Text: 2,
    *: 0
  }
}
```

**Lua equiv**: 用 `__type` 字段模拟 brand tag，`if .. elseif .. else` 替代 match。
**Analytic expected**: `n` 偶数时 `n * 3 / 2` ((1+2)/2 平均)，n=10000 → 15000。
**Backends**: tree_walk + luajit + (可能) bytecode（match lowering 已在 bytecode IR 中）；wasm/llvm_aot 走 paper-win gate（match 跨 funcref 边界尚未在 Z.4 envelope）。

### W22_ref_resolution — `&root` / `&sibling` 路径解析 — **TRUE BLOCKER, 暂搁**

**原本目的**：测 `&ref` walking + path-cache 命中。fixtures/references.relon / path_caching.relon 是直接 source。

**Blocker (Phase 1 agent #341, 2026-05-29 audit)**：

任何把 per-iter `&root` / `&sibling` / `&uncle` deref 放进 reduce closure 的 source 都被
evaluator 的 circular-reference guard 拒（`crates/relon-evaluator/src/reference.rs:777-786`）：
`result: range(n).reduce(...)` 评估时，owning dict 的 path 在 `evaluating_paths` 中，闭包
内任何 `&root` 都回到同一 dict 触发循环。

唯一能跑通的形态 `cfg_ref: &sibling.cfg` 把 ref pre-resolve 到 `Value::Dict` alias，
但 reduce body 读 `cfg_ref.port` 跟 W13_deep_dict_access 走的是同一路径 —— 没有 per-iter
Reference codepath / path-cache probe。booking 在 `W22_ref_resolution` label 下违反
/perf Honesty Rules "same code path" 关。

**解决路径**: evaluator-side fix —— 放宽循环 guard，允许 ref 指向**已封闭** sibling subtree
(owning dict 自身在 evaluating 但 target subtree 已物化完成)。Phase 1 agent 列为 TODO，
W22 暂 defer 到该 fix 落地后重启。任务 #341 描述已更新。

**Backends**: 当前 0 row。fix 落后预期 tree_walk + luajit 至少。

### W23_dict_spread — `{ ...&sibling.base, key: val }`

**目的**：测 dict spread 的 copy 成本（spread 是配置 merge 的主路径，fixtures/data_structures.relon 直接形态）。

**Production source (Relon)**：
```relon
#unstrict
#main(Int n) -> Int
{
  #internal
  base: { a: 1, b: 2, c: 3, d: 4 },
  result: range(n).reduce(0, (acc, _) =>
    acc + len({ ...&root.base, e: 5 }))
}
```

**Lua equiv**: 每 iter 用 table 复制 base + 加新 key。
**Analytic expected**: `n * 5`，n=10000 → 50000。
**Backends**: tree_walk + luajit。dict spread codegen 在 wasm walker 待定。

### W24_list_comprehension — `[x for x in xs if pred]`

**目的**：测 list comprehension lowering 路径，与 `.filter().map()` 直接对比同语义不同语法。

**Production source (Relon)**：
```relon
#unstrict
#import list from "std/list"
#main(Int n) -> Int
list.sum([x * 2 for x in range(n) if x % 3 == 0])
```

**Lua equiv**: `for x = 0, n-1 do if x%3==0 then s = s + x*2 end end`
**Analytic expected**: `2 * sum(x : x ∈ [0,n), x%3==0)`；n=10000 → 计算解析常数。
**Backends**: tree_walk + bytecode + luajit + wasm (若 walker 支持 comprehension lowering)。

### W25_pipe_chain — `range(n) | map(...) | filter(...) | sum()`

**目的**：测 pipe operator codegen，与 `.map().filter().sum()` 链式 mvp 对比。

**Production source (Relon)**：
```relon
#unstrict
#import list from "std/list"
#main(Int n) -> Int
range(n) | list.map((x) => x + 1) | list.filter((x) => x % 2 == 0) | list.sum()
```

**Lua equiv**: 标准 loop 同算法。
**Analytic expected**: 偶数 (x+1) 之和，对 n=10000 解析常数。
**Backends**: tree_walk + bytecode + luajit；wasm 视 walker pipe lowering 支持。

### W26_fstring_interp — `f"{a} ${b * 2}"`

**目的**：测 f-string 插值的 concat hot path（log/error 场景）。

**Production source (Relon)**：
```relon
#unstrict
#main(Int n) -> Int
range(n).reduce(0, (acc, i) => acc + _len(f"item ${i} of ${n}"))
```

**Lua equiv**: `string.format("item %d of %d", i, n)` + `#str`。
**Analytic expected**: `n` 个 string 的 byte length 之和；解析常数对 n=1000 可算。
**Backends**: tree_walk + luajit；bytecode/wasm 视 f-string lowering 支持。

### W27_stdlib_dict — `std/dict` map / filter / merge

**目的**：测非 list stdlib 模块的 import + 函数派发覆盖（13/15 row 都 only 用 std/list）。

**Production source (Relon)** (假设 stdlib 有 `std/dict.keys/values/len`)：
```relon
#unstrict
#import dict from "std/dict"
#main(Int n) -> Int
range(n).reduce(0, (acc, _) =>
  acc + dict.len({ a: 1, b: 2, c: 3 }))
```

**Lua equiv**: 手写 table + len。
**Analytic expected**: `n * 3`。
**Backends**: 视 std/dict 模块实际提供的函数。先 audit stdlib 表面再决定具体 method。

### W28_float_mixed_ops — float div / mod / Int→Float 转换

**目的**：补齐 W2/W20 之外的 float 算子覆盖（div、mod、混合 Int+Float、Float→Int trunc）。

**Production source (Relon)**：
```relon
#unstrict
#main(Int n) -> Float
range(n).reduce(0.0, (acc, i) => acc + i / 3.0 + i % 7)
```

**Lua equiv**: Lua 5.4 `//` 与 `/` 区分；要保持语义一致。
**Analytic expected**: 解析常数 (sum of i/3.0 + i%7 over i ∈ [0,n))。
**Backends**: tree_walk + bytecode + luajit；wasm/llvm_aot 需 f64 lowering 支持。

### W29_null_coalesce — `x ?? default` 空值回退

**目的**：测 null 路径 + null-coalesce 操作符，是 JIT 类型护栏的语义前提。

**Production source (Relon)**：
```relon
#unstrict
#main(Int n) -> Int
range(n).reduce(0, (acc, i) => acc + (i % 5 == 0 ? null : i) ?? -1)
```

**Lua equiv**: `if x == nil then x = -1 end`。
**Analytic expected**: 解析 (sum of i if i%5≠0 else -1 over i ∈ [0,n))。
**Backends**: tree_walk + luajit；bytecode 视 null/?? IR 支持。

### W30_strict_mode_baseline — `#strict` 默认模式（无 `#unstrict`）

**目的**：测 strict 模式分析器路径下的代表 workload（与 W6 unstrict 同算法但走严格类型检查）。

**Production source (Relon)**：
```relon
#main(Int n) -> Int
list.sum(range(n).map((Int i): Int => i + 1))
where {
  #import list from "std/list"
}
```

**Lua equiv**: 同 W6_lua。
**Analytic expected**: `n*(n+1)/2`，与 W6 同。
**Backends**: 与 W6 同 (tree_walk + bytecode + wasm + wasm_fast + luajit)；可与 W6 直接对比测分析器 / IR 在 strict vs unstrict 上的开销差。

## 4. 风险 / 已知约束

- **brand/match codegen 范围**: 当前 wasm walker (Z.4) 不支持 brand tag + funcref 跨 module，W21 预期只能 tree_walk + luajit + 可能 bytecode 出 row。若 bytecode 也拒，则单 row + 单 baseline 对比意义有限——需明确"这是覆盖目标而非速度对比"。
- **&ref path-cache**: W22 需检验 path-cache 是否在 production source 编译路径上启用；若是 IR-level constant fold 走掉，需 `black_box` 或动态 key 注入避免 fold。
- **stdlib std/dict 表面**: W27 实施前需先 grep `crates/relon-stdlib` 看导出函数清单，避免规划用不存在的 method。
- **strict 模式 source 形态**: W30 需保证类型注解严格 (`(Int i): Int => ...`)，避免 fallback 到 unstrict。
- **paper-win gate 覆盖**: 任何 ratio 看起来 "Relon 远超 LuaJIT" 的 row 都要走 timing-math / closed-form 三关 audit，必要时加 gate 或 rename。
