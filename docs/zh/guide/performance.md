# 性能与执行后端

> 本页**面向使用者**：用尽量通俗的语言介绍 Relon 是如何把脚本跑得又安全又快的，不需要懂编译原理。
> 如果你想看实现细节，参考 [架构概览](./architecture.md)。

## 一、Relon 是什么

Relon 是一种**可编程配置语言**：你写一段类似 JS / Lua 的代码，告诉系统"这个用户能不能登录"、"这个订单要不要打折"、"这条日志要不要丢弃"。宿主程序拿到代码后**执行它**，得到 yes / no 或一个数字。

执行得越快越好。LuaJIT 是这类语言的传统"速度王者"——Relon 的目标是在保留**沙箱安全**的前提下追平甚至超过它。在下文列出的基准上，Relon 的编译后端已全面快于 LuaJIT（2.7×–17×），并与原生 Rust 同级。

## 二、三个执行后端

把执行方式想成**饭店做菜**：

| 后端 | 做法 | 比喻 | 编译成本 | 稳态执行 |
|---|---|---|---|---|
| **Tree-walk**（树遍历解释器） | 每次执行都直接遍历语法树 | 路边小馆，每单去翻菜谱 | 零 | 慢 |
| **Cranelift AOT**（提前编译） | 把代码编译成本机机器指令 | 中央厨房，常用菜预制 | 百微秒级 | 快 |
| **LLVM AOT**（LLVM 提前编译） | 走 LLVM 优化管线生成机器码 | 米其林后厨，备菜最久 | 较高 | **最快**（与原生 Rust 同级） |

越往下稳态越快，但**准备成本越高**。三个后端跑的是同一份 IR、同一套沙箱语义，结果按位一致。Tree-walk 覆盖全部语言面；编译后端覆盖一个明确声明的支持面（见下文 Auto 的回退说明）。

## 三、`Backend::Auto` 怎么选档

你**不需要手动指定**后端——SDK 默认入口 `Backend::Auto` 的选档规则很简单：

1. **平凡标量短路**：如果程序是一个平凡的标量 `#main`（一个 AST 分类器在毫秒内判断），直接走 tree-walk——这类程序解释一次比编译一次更便宜。这是**性能短路**，不是能力回退。
2. **其余情况惰性编译**：第一次真正执行时才编译为 Cranelift AOT，编译结果在进程内缓存（`OnceLock`），同一程序后续调用零编译成本。

`Backend::Auto` 还会**按能力自适应**：若程序用到编译后端（Cranelift AOT）目前编不出的 `#main` 形状（例如返回 `#main() -> List<P>`），auto 会透明退回 tree-walk 解释器，仍产出正确结果——并记一条日志（让你知道这次运行跳过了 AOT 加速），而不是直接报错。真正的源码错误与宿主故障**不会**被吞掉，照常浮现。

想强制指定时，`EvaluatorBuilder` 提供 `Backend::TreeWalk`、`Backend::CraneliftAot`、`Backend::LlvmAot` 三个显式选项。

## 四、Sandbox（沙箱）是什么——为什么慢一点也得保

Relon 跑用户写的配置，**不能让用户代码搞坏宿主程序**。所以每个操作都有 4 道安检：

| 安检 | 防什么 | 比喻 |
|---|---|---|
| **Bounds check**（边界检查） | 不准访问数组之外的内存 | 不准翻墙看邻居家 |
| **Trap**（陷阱） | 算术溢出 / 除以零必须报错 | 仪表盘红灯一闪就熄火 |
| **Capability gate**（权限闸门） | 不准调用没授权的功能 | 只让你用授权过的钥匙 |
| **Resource limit**（资源上限） | 不准死循环吃光 CPU / 内存 | 限时 1 分钟，超时强制下线 |

LuaJIT **没有**这些安检，所以它的速度是"裸奔"基线。Relon 在保留全部安检的前提下追平甚至反超——这是设计上的核心成果。

详见 [沙箱与权限](./sandbox.md)。

## 五、当前性能现状（2026-06-10，W 系列基准）

**测量方法**：固定基准机（Xeon E5-2620 v4，Broadwell-EP），`taskset -c 2` 绑核，criterion 100 样本 × 5 s 测量窗口，空载（load1 ≈ 0）。下表为整条求值路径的耗时（越小越快），对照组为手写原生 Rust 实现与 LuaJIT 2.1：

| 基准 | 规模 | Tree-walk | Cranelift AOT | LLVM AOT | 原生 Rust | LuaJIT |
|---|---|---|---|---|---|---|
| 递归 fib | n=22 | 123.94 ms | — | 85.872 µs | 84.997 µs | 898.10 µs |
| 快速排序 | 1000 元素 | 105.67 ms | 681.76 µs | 78.130 µs | 110.48 µs | 972.19 µs |
| 二分查找 | 100 次 | 3.6072 ms | 13.287 µs | 2.4508 µs | 2.2840 µs | 6.0968 µs |
| 素数筛 | n=10000 | 491.87 ms | 3.4018 ms | 752.28 µs | 751.60 µs | 2.7049 ms |
| 矩阵乘法 | 16×16 | 26.211 ms | 41.535 µs | 9.5944 µs | 9.4295 µs | 43.299 µs |

读这张表的几个要点：

- **LLVM AOT 与原生 Rust 同级**：四个基准在 ±2% 内持平。注意诚实前提——Relon 的 LLVM AOT 默认按本机 CPU（`target-cpu=native`）编译，而 rustc 默认按通用 x86-64 编译；对照组 Rust 以**相同 target-cpu**（broadwell）重编后才能公平互比，上表的同级结论正是在这个前提下得出的（对照实验中仅二分查找一项 Rust 快 3.2%，其余为持平）。
- **快速排序快过 Rust 1.41×** 不是算法差异：两边算法一致，差异来自分配策略——Relon 编译后端用 scratch bump-arena 分配列表，Rust 对照实现走 malloc。这是分配器红利，不是"语言更快"。
- **全部五个基准快于 LuaJIT 2.7×–17×**（LLVM AOT 口径），且 Relon 全程保留 4 道沙箱安检，LuaJIT 没有。
- **Cranelift AOT** 比 LLVM 慢 4×–6×，但编译成本低两个数量级，适合冷启动敏感场景——这正是 `Backend::Auto` 默认选它的原因。
- Tree-walk 慢 3–4 个数量级是预期内的：它是覆盖全语言面的参照实现与回退兜底，不是性能路径。

## 六、行口径与核口径——两套数字不能互比

跨语言边界调用有固定的**封送成本**（参数、返回值在宿主与沙箱表示之间转换）。同一个最小基准（W12 标量内核）：

- **行口径**（含每次调用的封送）：约 192 ns / 次；
- **核口径**（纯计算内核，扣除封送）：3.38 ns / 次，与原生 Rust 内核（3.40 ns）持平。

差值约 185 ns 就是每次跨界调用的封送成本。**两套口径严禁互相比较**——拿自己的核口径去比别人的行口径（或反之）都是错误结论。如果你的场景是高频小调用，封送成本是主项，应把多次计算合并进单次调用。

## 七、Cached cold start——重启不用重新编译

Relon 把 Cranelift 编译产物缓存到本地：IR 缓存（跳过 parse + analyze + lower 的快速恢复路径）已在冷启动中生效；链接好的 `.o` 对象缓存（ELF 字节 + 完整性元数据）**已接通 dlopen 直接执行**——同一份源码的第二次冷启动命中缓存后，按 *HMAC 校验 → 生成器版本核对 → dlopen 加载 → 符号解析 → 执行* 的链路直接运行缓存的机器码，完全跳过 parse + analyze + lower + codegen。链路上任一环节失败（文件损坏、版本漂移、符号缺失）都会响亮记日志、作废过期条目并退回进程内编译——程序行为不变，且缓存命中的运行结果与新鲜编译按位一致。

缓存带 HMAC-SHA256 完整性校验（防篡改 + 防第三方注入），key 存在 `$XDG_DATA_HOME/relon/cache-key`（mode 0600）。HMAC 校验永远在 dlopen **之前**进行——认证失败的对象绝不加载。没有可用 key 时缓存读写都会被拒绝并降级为正常冷启动编译。生成器版本不匹配（缓存写入后编译器已变更）按 miss 处理：重新编译并覆写该条目。

## 八、一句话总结

> **解释器保证全覆盖与正确性，编译后端负责速度；`Backend::Auto` 自动选档，编不了就响亮地退回解释器——快的路径与慢的路径产出按位一致的结果。**

这就是 Relon 在保留完整沙箱的前提下追平原生 Rust、反超 LuaJIT 的全部路线。
