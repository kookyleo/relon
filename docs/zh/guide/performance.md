# 性能与执行档位

> 本页**面向使用者**：用尽量通俗的语言介绍 Relon 是如何把脚本跑得又安全又快的，不需要懂编译原理。
> 如果你想看实现细节，参考 [架构概览](./architecture.md)。

## 一、Relon 是什么

Relon 是一种**可编程配置语言**：你写一段类似 JS / Lua 的代码，告诉系统"这个用户能不能登录"、"这个订单要不要打折"、"这条日志要不要丢弃"。宿主程序拿到代码后**执行它**，得到 yes / no 或一个数字。

执行得越快越好。LuaJIT 是这类语言的"速度王者"——我们的目标是在保留**沙箱安全**的前提下，把 Relon 追到同一水平。

## 二、四种执行档位（速度阶梯）

把执行方式想成**饭店做菜**：

| 档位 | 做法 | 比喻 | 启动 | 执行 |
|---|---|---|---|---|
| **Tree-walk**（树遍历） | 每次执行都从头看一遍菜谱 | 路边小馆，每单去翻菜谱 | 即时 | 慢 |
| **Bytecode VM**（字节码虚拟机） | 把菜谱预先翻译成"操作清单" | 中餐厅，菜谱拆成步骤卡 | 毫秒 | 中 |
| **Cranelift AOT**（提前编译） | 把代码编译成机器指令 | 中央厨房，常用菜预制 | 百微秒 | 快 |
| **Trace JIT**（热点追踪即时编译） | 观察哪条路径最热，专门给它生成最快机器码 | 美食流水线，热门套餐有专用工位 | 触发后 | **极快** |

越往下越快，但**准备成本越高**。Relon 会**自动选档**：冷门代码用 tree-walk（启动快），热门代码自动升到 trace JIT（执行快）。

你**不需要手动指定**档位——SDK 入口 `Backend::Auto` 会按 IR op 数量阈值动态切换。

## 三、Trace JIT 的核心魔法

Trace JIT 是性能的**最高档**，原理像**录像 + 剪辑**：

1. **观察**：先用 tree-walk 跑代码，给每个分支挂个计数器
2. **触发**：发现某条路径跑了一万次（"热点"）→ "这条值得优化"
3. **录像**：再次跑这条路径时，把每一步的操作记下来（叫 trace）
4. **剪辑**：删冗余、合并相邻操作、提前算好常量
5. **生成机器码**：把剪辑后的 trace 编译成 CPU 直接能跑的指令
6. **守卫（guard）**：每个录像都基于"假设"（输入是整数、数组够长等），代码里插 guard——一旦假设不成立，立刻**跳回慢路径**（叫 deopt）重新跑

类比：短视频 app 的**预加载**——你看哪个频道多，它就专门优化哪个频道；你突然换台，它退回普通模式。

## 四、Sandbox（沙箱）是什么——为什么慢一点也得保

Relon 跑用户写的配置，**不能让用户代码搞坏宿主程序**。所以每个操作都有 4 道安检：

| 安检 | 防什么 | 比喻 |
|---|---|---|
| **Bounds check**（边界检查） | 不准访问数组之外的内存 | 不准翻墙看邻居家 |
| **Trap**（陷阱） | 算术溢出 / 除以零必须报错 | 仪表盘红灯一闪就熄火 |
| **Capability gate**（权限闸门） | 不准调用没授权的功能 | 只让你用授权过的钥匙 |
| **Resource limit**（资源上限） | 不准死循环吃光 CPU / 内存 | 限时 1 分钟，超时强制下线 |

LuaJIT **没有**这些安检，所以它的 1 ns/iter 是"裸奔"基线。Relon 要在保留安检的前提下逼近它——这是设计上的核心权衡。

详见 [沙箱与权限](./sandbox.md)。

## 五、当前性能现状（截至 v6-δ M1）

下表是**真实测得**的 hot loop 数字（紧凑 `acc += i` 整数循环，越小越快）：

| 档位 | 时间 / iter | vs LuaJIT 1-3 ns/iter |
|---|---|---|
| Tree-walk | 2245 ns | 750-2245× |
| Cranelift AOT 暖调用 | 390 ns | 130-390× |
| Trace JIT 暖调用（v6-δ M1） | **9.52 ns** | 3-9× 慢 |
| LuaJIT 参考 | 1-3 ns | 基准 |

也就是说：

- 普通配置脚本（短代码、冷启动）→ tree-walk 即可，几微秒级别足够
- 中等复杂度（持续执行）→ AOT 编译一次，亚微秒响应
- 高频流处理（每秒百万次）→ trace JIT 进入 LuaJIT 同 class，已具备 ETL 流水线能力

## 六、Cached cold start——重启不用重新编译

Relon 把编译产物（`.o` 文件）缓存到本地，下次启动直接 dlopen，避免重复编译。当前 **339 μs cached cold start**（vs LuaJIT 同等架构同级别）。

缓存带 HMAC-SHA256 完整性校验（防篡改 + 防跨机迁移），key 存在 `$XDG_DATA_HOME/relon/cache-key` mode 0600。

## 七、为什么这条路靠谱

LuaJIT 用 20 年验证的就是**"观察热点 → 录像 → 编译 → guard 兜底"**这条路。Relon 的差异是：

- **必须保 sandbox**（LuaJIT 不用）→ 多了 guard hoisting 的设计成本
- **不用纯 JIT**（必须支持 AOT cache + 序列化执行）→ trace JIT 是 AOT 之上的"加速层"
- **正确性不能让**（生产环境跑用户配置）→ deopt 慢路径必须 100% 正确，trace 快路径只是"快路"

trace 快路径错了不要紧，慢路径接住；但慢路径错了就是 bug。所以**慢路径**（bytecode VM + 真 partial-resume）是性能优化的**地基**——把它修牢了，快路径才敢激进。

## 八、一句话总结

> **先观察哪条路最热，给它生成专用快路；快路靠不住时随时退回慢路，慢路保证 100% 正确。**

这就是从 tree-walk 到 LuaJIT trace-tier 同级的全部诀窍。
